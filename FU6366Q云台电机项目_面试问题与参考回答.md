# FU6366Q 云台电机项目：面试问题与参考回答

本文档用于准备简历中“云台姿态执行机构电机驱动控制固件”相关项目的面试追问。相比简历描述，面试时要主动讲清楚三件事：

- 这个项目不是简单调 demo，而是在原厂 FOC 框架上封装出云台主控可调用的电机执行器接口。
- 固件里存在三层关键状态机：电机控制状态机、传感器校准状态机、应用控制状态机。主要改动集中在应用层状态机和 CAN 应用层。
- 所有功能都要用遥测数据闭环验证：命令是否发出、固件是否收到、控制模式是否切换、目标值是否变化、电流是否输出、故障是否触发。

## 一、项目整体与职责边界

### 1. 这个项目最终要解决什么问题？

参考回答：

项目面向云台姿态执行机构，目标是把 FU6366Q 电机驱动板做成一个可由主控芯片通过 CAN 调用的电机执行器。它不是只让电机转起来，而是要支持云台实际需要的接口：定速转动、转到指定角度、当前位置保持、主动刹停、受扰后回正，以及通信异常时自动进入安全状态。

我做的工作可以概括为：在原厂 FOC 控制链路上实现电流、速度、位置模式的切换和调试；设计 CAN 控制协议，把 speed、position、hold、brake_hold 等能力抽象成主控可调用的命令；实现应用层运动状态机和安全保护；开发 Python 上位机与日志分析流程，用遥测数据验证控制效果。

### 2. 你到底改了哪些层？哪些是原厂已有的？

参考回答：

这个问题很关键。固件里大体可以分三层：

- `MC_Control()`，在 `FocControl.c`，是原厂电机控制主状态机，负责电机从 ready/init/run/fault 等状态切换。我基本没有重写这一层。
- `MR_Control()`，在 `SensorFunction.c`，是传感器/MR 校准状态机，负责上电辨识和角度校准。我也基本沿用原厂逻辑。
- `ApplyFun_Conrtrol()`，在 `ApplyFunction.c`，是应用层状态机，是我主要修改的地方。它决定在电机已经 ready 之后，根据按键或 CAN 命令选择电流、速度、位置、刹停、保持等应用行为。

除此之外，我还改了 `ControlChat.c` 做 CAN 应用层协议解析和遥测上报；改了 `CAN.c/CAN.H` 让底层 CAN 接收新的命令 ID；在 `AddFunction.c` 的 1ms tick 里挂接 CAN 超时计数。

所以面试时可以明确说：底层 FOC 框架和电机主状态机主要来自原厂，我的工作是在此基础上完成应用层控制状态机、CAN 协议封装、安全策略和上位机调试闭环。

### 3. 为什么要先用电脑模拟主控芯片？

参考回答：

最终系统中，主控芯片会通过 CAN 给 FU6366Q 发控制命令。如果一开始就用真实主控联调，问题来源会很多：姿态算法、主控 CAN 驱动、电机固件、电源、机械结构都可能干扰定位。

我先用 PC + PCAN-USB 模拟主控，可以把问题收敛到“CAN 协议 + 电机驱动固件”这条链路。PC 上位机能够手动下发命令并记录遥测数据，方便逐个验证 speed、position、hold、brake_hold 等接口。后续真实主控只要复用同一套 CAN 协议即可。

### 4. 简历里写“控制模式切换与调试”，具体体现在哪里？

参考回答：

具体体现在应用层把不同运动需求映射到 FOC 控制链路：

- Current 模式：设置 `mcFocCtrl.CtrlMode = CurrentLoopMode`，直接给 `mcFocCtrl.Iqref/FOC_IQREF`。
- Speed 模式：设置 `CtrlMode = SpeedLoopMode`，目标速度写入 `mcSpeedRamp.TargetValue`，速度环输出作为电流参考。
- Position 模式：设置 `CtrlMode = PostionLoopMode`，目标角度写入 `mcFocCtrl.TargetAngle.s32`，位置环输出速度参考，速度环再输出电流参考。
- BrakeHold：先进入速度模式，目标速度 0 rpm；停稳后切位置模式，锁定当前角度。
- Hold：直接进入位置模式，将当前位置作为目标角度。

这些模式在 CAN 命令中统一用 `mode` 字段表达，主控不用关心底层是几层环，只调用抽象接口。

## 二、三层状态机

### 5. 固件里有哪些状态机？你能画出整体执行关系吗？

参考回答：

可以这样理解主循环：

```text
while(1)
{
    GetCurrentOffset();    // 电流偏置校准
    MC_Control();          // 原厂电机控制状态机
    MR_Control();          // 原厂传感器/MR 校准状态机
    FUNC_Control();        // 应用层控制入口，内部调用 ApplyFun_Conrtrol()
    CAN_Process();         // CAN 接收命令和遥测上报
}
```

三层核心状态机是：

```text
MC_Control()          FocControl.c
  负责电机生命周期：ready/init/run/fault 等

MR_Control()          SensorFunction.c
  负责 MR 传感器校准、角度对齐、校准完成标志

ApplyFun_Conrtrol()   ApplyFunction.c
  负责应用行为：按键/CAN 命令、电流/速度/位置/刹停/保持/超时保护
```

只有当 `MRSyS.MRState == MRFinish && mcState == mcRun` 时，应用层才真正执行运动命令。

### 6. 为什么前两个状态机你不大改？

参考回答：

因为 `MC_Control()` 和 `MR_Control()` 属于电机驱动底层基础流程，涉及电流采样、FOC 启动、传感器校准、保护状态等，原厂 demo 已经提供了相对完整的流程。我的目标不是重写底层电机启动框架，而是在原厂框架 ready 之后封装云台需要的应用接口。

如果随意改 `MC_Control()` 或 `MR_Control()`，很容易引入启动失败、角度不准、保护误触发等问题。因此我主要把改动集中在应用状态机 `ApplyFun_Conrtrol()` 和 CAN 协议层，这是风险更可控、也更贴近项目目标的做法。

### 7. 应用状态机 `ApplyFun_Conrtrol()` 做了什么？

参考回答：

它主要负责把外部命令转化为控制模式和目标值。当前支持：

- 上电 ready 后默认 Hold 当前角度；
- 按键 fallback：无 CAN 命令时，按键可以低速点动和刹停保持；
- CAN Current：直接电流模式；
- CAN Speed：速度模式；
- CAN Position：位置模式；
- CAN BrakeToHold：先 0 rpm 刹停，停稳后锁角；
- CAN Hold：锁住当前位置；
- CAN Disable：0 电流输出；
- CAN 超时：连续命令超时后自动刹停保持。

应用状态机本质上是把“云台动作语义”映射到 FOC 的 `CtrlMode`、`TargetAngle`、`mcSpeedRamp.TargetValue`、`Iqref` 和限幅参数。

### 8. 为什么要等 MRFinish 和 mcRun？

参考回答：

FOC 控制依赖角度反馈、电流采样偏置、传感器校准结果。如果 MR 角度没校准完成，位置控制的目标角度和实际角度就不可靠；如果电机主状态机没进入 `mcRun`，底层 PWM/FOC 可能还没准备好。

所以应用层用：

```c
MRSyS.MRState == MRFinish && mcState == mcRun
```

作为 ready 条件。CAN 命令可以先收到，但不应该在底层未 ready 时驱动电机。

## 三、FOC 控制环逻辑

### 9. 三层控制环的物理结构（由内到外）
在FOC（磁场定向控制）中，这三层是级联嵌套的
- 最内层：电流环（Iq/Id 环） —— 执行层
- 中间层：速度环（Speed 环） —— 指导层
- 最外层：位置环（Position 环） —— 决策层
它们的信号流向是：位置环 → 速度环 → 电流环 → PWM输出 → 电机

1、电流环（最内层，底层硬件级）：最底层、最硬核的环节。它直接控制流入电机线圈的电流（Iq 和 Id）。它接收一个“目标电流值”（Iq_ref），然后实时采样电机相电流，通过PID算法调整PWM占空比，强迫实际电流（Iq_actual）紧紧跟随目标值。

在项目中的体现：文档里提到的 FOC_CR2 的 UQD/UDD 位，就是用来把FOC硬件强制锁定在“电流输出模式”的。如果这层没生效（比如之前被速度环覆盖），现象就是：iq_ref_raw 有值，但 iq_actual_raw 接近0，电机不出力。JOG_IQ_LIMIT_A 和 HOLD_IQ_LIMIT_A（0.5A~0.6A）就是套在这层外面的“紧箍咒”，防止电流过大烧电机。

2、速度环（中间层，算法级）：控制电机转动的快慢。它接收一个“目标转速”（target_speed），实时检测当前实际转速（speed_rpm_est），计算差值（SpeedErr）。速度环PID根据这个差值，计算出一个对应的电流指令（即 Iq_ref），然后交给内层的电流环去执行。

在在项目中的体现：
- 按键点动（JOG_SPEED_RPM=50）时，固件调用 Apply_ConfigSpeedOutput，设置速度环的目标值。
- 松手刹停时，把目标速度设为 0。此时速度环会拼命把速度往0拉，这个“拼命”的力度就体现在输出的 Iq_ref 上（受 JOG_IQ_LIMIT_A 限制）。

3、位置环（最外层，应用级）：控制电机转到哪个角度并停住。它接收一个“目标角度”（target_angle），实时检测当前实际角度（actual_angle），计算差值（angle_error）。位置环PID根据这个角度差，计算出一个对应的速度指令（即目标转速），交给中间的速度环去跑，速度环再翻译成电流给电流环。

在项目中的体现：上电锁定时，调用 Apply_HoldCurrentAngle，把当前角度设为 target_angle。当外力拧动它时，angle_error 变大，位置环输出一个纠偏转速，电机就自动转回去。

三种工作模式（Ctrl_Mode）与三层环的关联:
工作模式 (Ctrl_Mode)	            |   接入层级	                                          |                                      项目中的应用场景
1. CurrentLoopMode（电流/力矩模式）|	只启用电流环	用户直接给 Iq_ref ➔ 电流环 ➔ 电机。     |              底层诊断、直接测力矩。文档里直接电流诊断版（按键±0.3A让电机猛转）就是这个模式。不管速度和角度。
2. SpeedLoopMode（速度模式）	    | 启用速度环 + 电流环	用户给目标速度 ➔ 速度环（算出Iq）➔  |
                                 |     电流环 ➔ 电机。                  |位置环被旁路。    |                   	日常匀速转动、云台连续跟踪。CAN命令里的 Speed 和 Brake Hold（目标速度=0）都在此列。
3. PositionLoopMode（位置模式）   |	全启用（位置环 + 速度环 + 电流环）	用户给目标角度 ➔     |
                                 | 位置环（算出目标速度）➔ 速度环（算出目标电流）                                静止保持、抗扰动锁定、点对点转动。CAN命令里的 Position 和 Hold 都是此模式。

                                 | ➔ 电流环 ➔ 电机。                                                                
### 9. 你理解这个项目里的控制环结构吗？

参考回答：

理解。这个项目里可以看成三层控制：

```text
位置环：TargetAngle - SensorAngle -> PostionLoopOut，输出速度参考
速度环：目标速度 - SpeedFlt -> PI2，输出 Q 轴电流参考
电流环：Iqref - IqFeedback -> 硬件电流 PI/PWM
```

电流模式只使用最内层电流环；速度模式使用速度环 + 电流环；位置模式使用位置环 + 速度环 + 电流环。

### 10. CurrentLoopMode 是怎么走的？

参考回答：

CurrentLoopMode 下，应用层直接设置：

```c
mcFocCtrl.CtrlMode = CurrentLoopMode;       // 状态机管理层
mcFocCtrl.CurrentLoopEn = 1;                // 软件算法层（PI调节器使能）,在中断里，会判断这个标志位来决定是否调用软件电流环PI调节器
mcFocCtrl.Iqref = iq_ref;                   // 软件算法层（目标值缓存）,给日志和遥测（0x181帧）看的。这个变量会被打包发到上位机
FOC_IQREF = iq_ref;                         // 硬件寄存器层（硬件加速引擎的输入）,写给硬件寄存器的最终执行命令。物理PWM发生器只认这个地址的值。
```

FU6366Q的主控芯片（FU68xx系列）内部有一个专门做FOC（磁场定向控制）的硬件加速引擎。FOC_CR2 是这个硬件引擎的控制寄存器2。
UQD/UDD 状态                     硬件FOC引擎的工作模式                               效果   
SetBit（置1）                     电压模式（开环/前馈）             硬件忽略软件给的 Iqref（电流指令），直接根据 Uq（电压指令）输出PWM。这时候电流环是物理断开的，电机表现为“给定电压，转速随负载变”。
ClrBit（清零，即“强制打开”）       电流模式（闭环）                 硬件强制使用软件给的 Iqref，实时采样相电流做闭环比较，输出对应的PWM。这时候电流环是物理接通的，电机表现为“给定电流，力矩恒定”。


并且需要确保 FOC 硬件处于电流模式：

```c
ClrBit(FOC_CR2, UQD);         // 关闭Q轴电压模式，强制进入电流模式
ClrBit(FOC_CR2, UDD);         // 同上，关闭D轴的电压模式
FOC_IDREF = 0;                // D轴目标电流强制为0
MOE = 1;
```
如果默认使用电流模式，
之前踩过一个坑：原始 `Speed_response()` 在 CurrentLoopMode 分支里设置了 `FOC_IQREF`，但没有 `return`，后面速度 PI 又覆盖了 `FOC_IQREF`。后来给 CurrentLoopMode 加了 `return`，避免直接电流命令被覆盖。

### 11. SpeedLoopMode 是怎么实现定速控制的？

参考回答：

应用层设置：

```c
mcFocCtrl.CtrlMode = SpeedLoopMode;
mcFocCtrl.CurrentLoopEn = 1;
mcSpeedRamp.TargetValue = S_Value(target_rpm);
mcSpeedRamp.FlagONOFF = 1;
s_PI.Uk_max/min = Iq 限幅;
```

在 `Speed_response()` 中，速度环计算：

```c
mc_ramp();
mcFocCtrl.SpeedErr = mcSpeedRamp.ActualValue - mcFocCtrl.SpeedFlt;
mcFocCtrl.UQValueTemp = HW_One_PI2(mcFocCtrl.SpeedErr);
FOC_IQREF = mcFocCtrl.UQValueTemp;
```

也就是说，速度 PI 根据速度误差输出需要的 Q 轴电流，电流环再驱动 PWM。

### 12. PostionLoopMode 是怎么实现定点控制的？

参考回答：

应用层设置：

```c
mcFocCtrl.CtrlMode = PostionLoopMode;
mcFocCtrl.CurrentLoopEn = 1;
mcFocCtrl.TargetAngle.s32 = 目标角度;
PI_Position.Uk_max/min = 位置环最大速度输出;
s_PI.Uk_max/min = 速度环最大电流输出;
```

1ms tick 中，如果处于位置模式，会调用 `Posi_response()`。位置环计算：

```c
Err = _Q15Limit(TargetAngle - SensorAngle);
PostionLoopOut = Kp * Err + Kd * (Err - ErrLast);
```

然后在 `Speed_response()` 中，位置环输出作为速度参考：

```c
mcFocCtrl.SpeedErr = mcFocCtrl.PostionLoopOut - mcFocCtrl.SpeedFlt;
```

后面仍然通过速度 PI 生成 `FOC_IQREF`。所以位置模式本质是“位置环给速度目标，速度环给电流目标”。

### 13. 为什么位置环输出速度，而不是直接输出电流？

参考回答：

位置环输出速度更容易控制运动过程。位置误差大时输出较高速度，接近目标时速度下降，再由速度环计算所需电流。这样可以自然限制最大运动速度，也更容易避免过冲和突变。

如果位置误差直接映射成电流，电流变化会更硬，机械响应容易抖动，限速也不方便。

### 14. Iq 和力矩有什么关系？

参考回答：

对永磁同步电机，在 `Id` 约等于 0 的情况下，Q 轴电流 `Iq` 主要决定电磁转矩。可以近似认为：

```text
Torque = Kt * Iq
```

厂家给的 Kt 约 `0.2 N·m/A`。比如 `Iq = 0.5A`，电机侧力矩约：

```text
0.2 * 0.5 = 0.1 N·m
```

所以限流不仅是保护电路，也是在限制最大输出力矩。

## 四、8051、定点 Q 格式与数值换算

### 15. 你项目里用到了 Q 格式吗？为什么？

参考回答：

用到了。FU6366Q 这类 8051 内核没有 FPU，浮点运算开销大，不适合在高频控制环里大量使用。因此原厂控制框架大量使用定点 Q 格式，比如 `_Q15`、`S_Value()`、`I_Value()`。

典型定义类似：

```c
#define _Q15(A) (int16)((A) * 0x7fff)
#define S_Value(Speed) _Q15(Speed / MOTOR_SPEED_BASE)
#define I_Value(Curr)  _Q15(Curr * Rshunt * Gain / Vref)
```

应用层输入是人容易理解的单位，比如 rpm、mA、角度。固件里要转换成 Q15 或角度 raw，再交给控制环。

### 16. 速度 rpm 是怎么转成内部值的？

参考回答：

原框架里速度用标幺化值表示，基准是 `MOTOR_SPEED_BASE`，例如 4000 rpm。宏类似：

```c
S_Value(rpm) = _Q15(rpm / MOTOR_SPEED_BASE)
```

所以 50 rpm 会转成一个较小的 Q15 数值，写入 `mcSpeedRamp.TargetValue`。遥测反算时用：

```text
rpm = speed_raw * MOTOR_SPEED_BASE / 32767
```

### 17. 电流 mA 是怎么转成 Iqref 的？

参考回答：

上位机协议里 current 和 limit 用 mA，固件里先转成 A，再通过 `I_Value()` 变成电流环使用的 Q15/硬件参考值。

之前也用过遥测换算：

```text
Iq_A = Iq_raw * 4.5 / (32767 * 0.005 * 8)
     = Iq_raw * 0.003433
Torque_Nm = Iq_A * 0.2
```

这个换算用于从 CSV 中把 raw 电流转成实际电流和估算力矩。

### 18. 角度是怎么表达的？为什么 Position 超过 180 度可能异常？

参考回答：

角度在内部常用 16bit 周期表示一圈，即：

```text
360° -> 65536
```

位置目标使用 `TargetAngle.s32`，实际角度是 `SensorAngle.s32`。位置误差会经过：

```c
_Q15Limit(TargetAngle - SensorAngle)
```

这会把误差限制在约 `±32767`，对应大约 `±180°`。因此单次相对位置命令如果超过 180°，控制环无法表达更长路径，表现上可能像都在转半圈附近。

所以 GUI 后来对 relative position 加了 `-180°..180°` 提示。大角度可以拆成多次命令，或使用速度模式转动。

## 五、CAN 控制协议

### 19. CAN 协议具体怎么设计？

一帧 CAN 数据可以理解成一个小信封,具体内容如下：
一帧 CAN 数据 =
{
    ID: 这封信是什么类型
    DLC: 数据长度
    Data[0~7]: 真正的数据内容，最多 8 字节
}
但是这 8 个字节到底代表什么，CAN 本身不知道，是代码自己规定的。比如规定：
0x180 表示电压、电流反馈
0x181 表示速度、温度、状态
0x200 表示控制命令
这就是所谓的应用层协议。

CAN 总线上的所有节点都能看到总线上的帧，但是每个节点会根据 ID 判断：“这帧是不是发给我处理的？我需要不要解析？我要不要忽略？”

遥测帧可以理解成 FU6366 把自己内部的运行状态发出来，包括电压、电流、转速、温度、角度、故障状态等。这里面不一定都是“寄存器里的值”，而是电机控制程序里的状态变量。其中有些来自硬件寄存器，有些来自 ADC 采样，有些是 FOC 算法计算出来的，有些是软件状态机变量。所以遥测帧真的本质是把 FU6366 当前运行状态打包，通过 CAN 发给上位机。

为什么我们用多个ID而不能用一个ID发送遥测帧数据呢？
因为 CAN 一帧最多只有 8 个数据字节。并且 CAN ID 不只是编号，它还代表总线仲裁优先级。理论上我们的遥测帧编号可以使任意的。但是要注意在 CAN 总线上：ID 数值越小，优先级越高；ID 数值越大，优先级越低。CAN ID 是我们自己定义的，只要在 0x000~0x7FF 标准帧范围内，理论上都能用。


参考回答：

我保留原来的遥测帧 `0x180~0x183`，新增：

```text
0x200：PC/主控 -> 电机控制命令
0x184：电机 -> PC/主控命令应答
```

`0x200` 是 8 字节标准帧：

| 字节    | 名称            | 类型    | 含义                   |
|---------|----------------|---------|------------------------|
| Byte0   | `mode`         | `uint8` | 控制模式                |
| Byte1   | `flags`        | `uint8` | enable、relative等标志  |
| Byte2~3 | `target`       | `int16` | 目标值，小端            |
| Byte4~5 | `limit_ma`     | `int16` | Iq电流限制，小端，单位mA |
| Byte6   | `timeout_50ms` | `uint8` | 超时时间，每1代表50ms    |
| Byte7   | `seq`          | `uint8` | 命令序列号              |

1. Byte0：mode
| mode | 模式 | target含义 |

| 0 | Disable | 固件切换到电流环并令：Iqref = 0;即零转矩输出 |
| 1 | Current | 目标Iq电流，单位mA，即Iqref = 300mA |

2. Byte1：flags
bit0：ENABLE
bit1：RELATIVE
| bit | 掩码 | 含义 |

target 是有符号16位整数：Byte2表示低8位、Byte3表示高8位

4. Byte4~5：limit_ma

5. Byte6：timeout_50ms

6. Byte7：seq

### 20. 为什么把命令设计成 mode + target + limit + timeout？

参考回答：

因为云台主控关心的是“我要电机做什么”，而不是底层控制环怎么实现。`mode` 表示动作语义，`target` 表示目标值，`limit` 表示安全边界，`timeout` 表示通信安全策略。

例如：

- speed：target 是 rpm，limit 是最大 Iq；
- position：target 是角度，limit 是最大 Iq；
- current：target 是 mA，limit 是电流上限；
- brake_hold：target 不重要，limit 控制刹停电流。

这样主控算法调用接口统一，固件内部再做模式切换和限幅。

1. Byte0：mode 表示固件当前记录的 CAN 命令模式：
2. Byte1：status 这是一个位状态字
| bit | 数值 | 名称 | 含义 |

### 22. 遥测帧怎么设计？

参考回答：

遥测帧共有4组：

```text
0x180：母线电压、电流采样、Iq/Id 实际值
0x181：Iqref、速度、温度、fault_state、mc_state
0x182：实际角度、目标角度、角度误差、Iq 峰值
0x183：目标速度、ramp 速度、按键状态、ctrl_mode、can_mode、can_status
```

这些帧覆盖了调试所需链路：电源/故障、控制模式、速度、电流、角度和 CAN 状态。

### 23. CAN 通信超时机制是怎么设计的？为什么超时时间从10ms单位改成50ms单位？Repeat Command又是什么？

在CAN控制协议中使用 0x200 作为控制命令帧。由于一帧标准CAN数据最多为8字节，Byte6只分配了1字节保存超时时间，因此其原始数值最大只能是255。最初我规定Byte6的单位为10ms：实际超时时间 = 255 × 10ms = 2550ms。所以测试时即使GUI填写5000ms，电机实际上仍然会在约2.5秒后触发超时保护。当速度模式或电流模式发生超时后（判断的不是CAN总线上有没有任意报文，而是有没有收到新的、有效的 0x200 控制命令。），应用状态机会：将目标速度设为0→ 使用速度环主动减速→ 实际速度低于约8rpm→ 记录当前位置→ 切换到位置环保持。这样即使主控死机、CAN断线或上位机退出，电机也不会继续执行最后一次速度或电流命令。并且我还在GUI中增加了Repeat Command ，其本质上是应用层心跳。也就是每50ms重新发送一次相同的控制命令。因此，只要GUI和CAN通信正常，就不会触发超时。如果GUI停止运行或CAN断开，固件超过所设置的超时时间没有收到新命令，就进入超时保护。我加入这种心跳模式就是因为电流模式和速度模式需要周期发送命令，而位置模式、刹停、抗绕保持等动作都只需要发送一次命令

### Repeat 和 CAN 硬件自动重发有什么区别？
CAN硬件自动重发用于解决单帧发送失败；Repeat是应用层周期生成新命令，用于持续更新控制目标并证明主控仍然在线。

## 六、功能实现细节

### 24. 定速控制功能怎么实现？

定速控制采用的是典型的 速度环加电流环串级控制。
上位机或主控通过CAN发送速度命令，命令中包含目标转速、最大允许电流、通信超时时间和序列号。固件接收到命令后，会先检查电机是否已经完成传感器校准并进入正常运行状态，然后对目标转速和电流进行限幅。
速度控制的基本流程是：输入目标转速
→ 速度斜坡处理
→ 与实际转速计算偏差
→ 速度PI控制器
→ 输出Iq电流参考值
→ 电流环控制
→ FOC产生三相驱动电压
→ 电机达到目标转速

其中，实际转速由位置传感器的角度变化计算得到。速度PI根据目标转速与实际转速之间的误差，动态调整Iq参考值。Iq主要决定电机转矩：
- 实际速度低于目标速度时，增加Iq，提高加速转矩；
- 实际速度高于目标速度时，减小Iq或输出反向Iq，使电机减速；
- 达到目标速度后，Iq主要用于抵消摩擦和负载扰动。

没有直接让目标转速瞬间跳变，而是利用速度斜坡逐渐调整内部速度给定，减少启动冲击、电流突增和机械振动。

### 为什么不能直接控制PWM实现定速？
PWM占空比和转速不是固定对应关系，会受到母线电压、负载和摩擦影响；采用速度闭环后，控制器可以根据速度误差自动调整转矩，因此负载变化时仍能维持目标转速。

### 25. 定点 Position 功能怎么实现？

参考回答：

CAN 收到 position 命令后，应用层：

- 将 `target` 从 0.01 度转换为内部角度 raw；
- 如果 `relative` 置位，就用 `SensorAngle + delta` 作为目标；
- 否则使用绝对角度目标；
- 设置 `CtrlMode = PostionLoopMode`；
- 设置位置环最大输出速度；
- 设置速度环 Iq 限幅；
- 写入 `TargetAngle.s32`。

后续由 1ms 位置环计算 `PostionLoopOut`，再由速度环生成 Iqref。

### 26. Hold 功能怎么实现？

参考回答：

Hold 是位置模式的一种特殊用法。收到 Hold 命令时：

```c
TargetAngle.s32 = SensorAngle.s32;
CtrlMode = PostionLoopMode;
```

也就是把当前实际角度作为目标角度。之后如果外力把电机扭开，`TargetAngle - SensorAngle` 会产生角度误差，位置环输出纠偏速度，速度环输出恢复电流，把电机拉回锁定位置。

### 27. 抗扰功能的原理是什么？

参考回答：

抗扰本质是位置闭环保持。静止时目标角度固定，外力扰动后实际角度变化，形成位置误差。位置环根据误差输出速度参考，速度环根据速度误差输出 Iqref，电流环产生恢复力矩。

我还对抗扰做了限幅：

- 位置环最大输出速度限制，比如 80 rpm；
- 速度环输出 Iq 限制，比如 500mA 或 600mA；
- 小误差死区避免微小抖动。

这样既能回正，又避免保持时电流过大或抖动。

### 28. BrakeHold 刹停功能怎么实现？

参考回答：

BrakeHold 分两段：

第一段是主动刹停：

```text
CtrlMode = SpeedLoopMode
TargetSpeed = 0 rpm
```

此时速度环会根据实际速度产生反向/阻尼电流，把速度拉到 0。

第二段是停稳锁角：

```text
if abs(SpeedFlt) <= lock_speed_threshold:
    CtrlMode = PostionLoopMode
    TargetAngle = SensorAngle
```

这样比直接 `Iqref=0` 更可靠，因为 `Iqref=0` 只是不给力，不会主动抵消惯性。

### 29. Disable 功能怎么实现？为什么后来改过？

参考回答：

最初 Disable 直接关掉 `mcSpeedRamp.FlagONOFF`，结果后续命令恢复不顺，出现 Disable 后按钮不好使的现象。

后来改成：

- 进入 CurrentLoopMode；
- `Iqref = 0`；
- `FOC_IQREF = 0`；
- 保持 `mcSpeedRamp.FlagONOFF = 1`；
- 控制链路保持活着。

也就是说 Disable 是 0 电流输出，不主动保持、不主动刹停，但后续 Speed/Hold/Position 命令可以恢复控制。常规停车建议用 BrakeHold，而不是 Disable。

## 八、数据分析与问题定位

### 34. 你怎么用数据分析控制问题？

参考回答：

我建立了一个链路式分析方法：

```text
tx 0x200 命令是否正确
rx 0x184 是否应答
0x183 ctrl_mode/can_mode/can_status 是否切换
0x182 target_angle/actual_angle 是否变化
0x181 iq_ref/speed/fault 是否正常
0x180 iq_actual 是否真的输出
```

这样可以逐层判断问题发生在上位机、CAN 接收、应用状态机、控制环还是硬件保护。

### 35. 你遇到过 `iq_ref` 有了但 `iq_actual` 没起来的问题吗？

参考回答：

遇到过。旧的位置保持版本中，`iq_ref_raw` 已经达到较大值，但 `iq_actual_raw` 长期接近 0，实际角度也几乎不动。这说明上层位置环已经要求出力，但底层没有真正输出力矩。

后来定位到：

- CurrentLoopMode 的 `FOC_IQREF` 被速度 PI 覆盖；
- FOC 硬件电流模式没有显式切入；
- 需要清 `UQD/UDD`、设置 `FOC_IDREF=0`、`MOE=1`。

修复后直接电流模式可以明显驱动电机，说明底层功率级和电流环可用。

### 36. 你怎么确认 Position 命令真的生效？

参考回答：

看 `0x182` 的目标角度和实际角度。曾经从 GUI raw log 中看到，Position 命令后目标角度从约 `321.5°` 变到 `21.5°`，实际角度随后追过去，相当于相对转了约 60°。

因此判断 Position 不是没生效，而是需要注意 relative 命令、repeat command 和单次 `±180°` 误差限制。

### 37. 为什么 pc_visualizer 有一次只生成表头没数据？

参考回答：

因为 GUI 已经占用了 `PCAN_USBBUS1`。同一个 PCAN 通道通常不能被两个 Python 程序同时可靠接收。`pc_visualizer` 打开了 CSV，但没有收到帧，所以只有表头。

后来我给 GUI 自己加了 raw CAN 日志，同时记录 tx 和 rx，不再需要同时打开另一个采集程序。

## 九、Python 上位机

### 38. Python 上位机做了什么？

参考回答：

上位机基于 Tkinter 和 python-can，主要功能是：

- 连接 PCAN-USB；
- 发送 `0x200` 控制命令；
- 显示 `0x180~0x183` 遥测；
- 显示 `0x184` 命令应答；
- 对 Current/Speed 周期续发；
- 对 Position/Hold/BrakeHold 单次发送；
- 记录 tx/rx raw CSV 日志。

它不是产品级界面，而是为了快速调试协议和控制链路。

### 39. 为什么 Current/Speed 要 repeat，Position 不 repeat？

参考回答：

Current 和 Speed 是连续命令。如果通信断开，电机不应该永远保持最后一个速度或电流，所以需要周期续发配合超时保护。

Position 是一次性目标命令，尤其 relative position 如果反复发送，会变成不断刷新“当前位置 + delta”，语义混乱。因此 Position 单次发送即可，执行过程由固件状态机和位置环完成。

### 40. 上位机日志怎么帮助面试中证明你做过调试？

参考回答：

我可以说日志里同时记录了：

```text
pc_time_s,direction,can_id,data_hex
```

其中 `tx 0x200` 是上位机发出的命令，`rx 0x184` 是固件应答，`rx 0x183/0x182/0x181` 是状态、角度、速度和电流遥测。通过这些数据可以复盘一次命令从 PC 到固件再到电机响应的全过程。

## 十、力矩估算与云台负载

### 41. 你怎么估算静态保持力矩？

参考回答：

用厂家给的力矩常数：

```text
Torque = Kt * Iq
Kt ≈ 0.2 N·m/A
```

如果保持电流限制为 0.5A，电机侧理论力矩约：

```text
0.2 * 0.5 = 0.1 N·m
```

如果有同步带减速，则输出端力矩：

```text
T_out = T_motor * ratio * efficiency
```

这个估算用于判断静态保持能力，但最终还要结合重心偏移、负载惯量、摩擦和实测电流峰值。

### 42. 连续运动力矩和静态保持力矩有什么区别？

参考回答：

静态保持主要抵抗重力矩和外部扰动。连续运动还要克服惯性力矩：

```text
T = J * alpha + load torque + friction
```

所以静态保持够用，不代表动态加减速够用。动态验证时要看加速段 Iq 峰值、速度跟踪误差和角度过冲。

## 十一、工程取舍与反思

### 43. 你最大的收获是什么？

参考回答：

最大的收获是电机控制调试必须数据闭环，不能只看代码逻辑。比如命令发了不代表电机执行了，目标角度变了不代表电流输出了，`Iqref` 有了也不代表 `Iq_actual` 跟上了。

我形成了按层排查的方法：命令 -> 应答 -> 状态机 -> 控制模式 -> 目标值 -> 实际值 -> 电流 -> 故障状态。

### 44. 如果重新做一次，你会怎么优化？

参考回答：

我会更早把协议语义分成连续命令和一次性命令；一开始就记录 tx/rx 全量 CAN 日志；更早加入命令范围提示，比如 position 单次 ±180°；同时把多电机节点 ID、CRC 和急停状态预留进协议。

### 45. 这个项目还有哪些没完成？

### 48. 为什么你说这是“电机控制抽象”？

参考回答：

因为主控算法不需要知道底层是电流环还是速度环，也不需要直接操作 `mcFocCtrl` 变量。主控只需要发：

```text
speed 50 rpm
position +30 degree
hold
brake_hold
```

固件内部负责把这些命令翻译成 FOC 控制模式、目标值、限幅和状态机切换。这个抽象层就是后续主控姿态算法和电机驱动之间的接口。

### 49. 如果电机抖动，你怎么调？

参考回答：

先判断抖动发生在哪个模式：

- Hold/Position 抖动：降低位置环最大速度输出、降低 Iq limit，检查位置环 Kp/Kd 和死区。
- Speed 抖动：检查速度 PI 参数、速度滤波、ramp 和电流限幅。
- Current 抖动：看电流采样和硬件电流环。

调参时不只看肉眼现象，要记录 `speed_raw`、`iq_ref`、`iq_actual`、`angle_error` 和故障状态。

### 50. 如果命令发了电机没动，你怎么排查？

参考回答：

按顺序查：

```text
1. tx 0x200 是否正确
2. rx 0x184 是否应答
3. fault_state/mc_state 是否正常
4. can_mode/ctrl_mode 是否切换
5. target_speed 或 target_angle 是否变化
6. iq_ref 是否变化
7. iq_actual 是否跟随
8. 电源、欠压、保护、硬件连接是否正常
```

这样可以很快判断问题是在通信、状态机、控制环还是硬件保护。
